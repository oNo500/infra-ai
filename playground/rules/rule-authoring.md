
# AGENTS.md 写作标准

本标准面向 AI 编码代理读取的规则文档(AGENTS.md / CLAUDE.md / .cursorrules 等)。
主要读者是 AI 代理,人类工程师可读但不优先。

本文档自身遵循其定义的规则,可作为参考样本。

---

## 1. 定位与加载

写之前先回答三个问题——它们决定文档应该多长、多严、放在哪。

### 1.1 加载时机

- **always-in-context**(根目录 AGENTS.md / CLAUDE.md):每次会话占用 token 预算,SHOULD 控制在 ~2K tokens 以内。经验上,中段规则的执行率随长度下降。
- **on-demand**(被代理工具按需检索):可以更长更详,但 MUST NOT 假定模型一定读到。

文档开头 SHOULD 标注其加载方式,让维护者知道边界。

### 1.2 分层

- 仓库根 AGENTS.md:全局约束、跨模块不变量。
- 子目录 AGENTS.md:局部约定(如 `frontend/AGENTS.md`、`infra/AGENTS.md`)。
- 局部规则 MAY 覆盖全局规则,但 MUST 显式声明覆盖关系。

### 1.3 与其他规则文件的关系

同一项目存在多份规则文件(`.cursorrules`、`copilot-instructions.md` 等)时,MUST 指定单一事实源,其余文件以引用指向它,避免漂移。

## 2. 写什么(三个排除测试)

按顺序应用以下测试,通过的内容才属于 AGENTS.md。

### 2.1 跨项目还成立?→ 不属于 AGENTS.md

跨项目通用的偏好(协作风格、何时提问、措辞习惯)属于 **system prompt 或个人配置**。
判据:换一个项目这条规则还成立,就不写。

### 2.2 工具能强制?→ 不属于 AGENTS.md

linter、formatter、type checker、test runner 能检查的,交给工具。AGENTS.md 只承载工具管不到的部分。
若必须在文档中提及,MUST 指向工具配置文件,而非重述规则。

### 2.3 临时状态?→ 放 issue tracker

"正在迁移到 X"、"等 Y 上线后改回" 这类内容 MUST NOT 写入规则文档——它们会过期且无人删除。

### 2.4 通过测试后剩下的内容

- **项目特异约束与不变量**(团队取舍、领域规则、安全/合规要求)。
- **覆盖语言/框架默认行为的约定**(上一类的常见子类)。
- **反模式**:历史踩过的坑(Things That Will Bite You)

## 3. 规则的组成

### 3.1 What:必须有

约束本身,一句话可复述。

### 3.2 Why:不显然时必须有

显然的安全/正确性规则可省 Why(如"密码 MUST NOT 写入日志")。
非显然的取舍 MUST 写 Why——经验上,缺少 Why 的取舍类规则在长上下文中容易被模型按默认行为覆盖。

### 3.3 How:必须可执行

给出判据或 canonical 示例。"写好测试" 不是 How;"每个公开函数 MUST 至少覆盖 happy path 与一个错误分支" 才是。

### 3.4 示例

✅ `日志 MUST 用 structlog——团队统一便于 ELK 解析。示例:log.info("payment_processed", user_id=u.id, amount=a)。`

(既已规定首选,无需再写"禁止 print / logging.*"——蕴含已足够,见 §5。)

## 4. 表达方式

### 4.1 行为强度:RFC 2119

| 关键词 | 用于 |
|---|---|
| MUST / MUST NOT | 违反导致 bug、安全问题或不可恢复后果 |
| SHOULD / SHOULD NOT | 违反只是不一致或低效,允许合理例外 |
| MAY | 真正可选 |

- SHALL 等同 MUST,本标准 SHOULD 统一用 MUST,避免双词并存。
- 滥用 MUST 会让模型对强制信号钝化——只在真正不可违反处用。

### 4.2 术语(ISO 704)

- **概念优先**:先明确要表达的概念,再找行业标准词,直接采用。
- **惯用术语优先**:不确定时停下查证(成熟领域查 Wikipedia,前沿查 RFC、主流仓库或源码)。MUST NOT 凭空造词。
- **中英文项目策略**:英文术语在工程语境广泛使用时,SHOULD 保留英文原词,不强译。
  - ✅ `idempotent` / `debounce` / `backpressure`
  - ❌ `重试可重入处理`(半英半中半生造)
- **约定词 vs 描述词**:目录树写"目录**约定**"而非"目录结构"——后者只描述当前状态,不构成约束。

### 4.3 示例优先于规则

对**风格、结构、措辞**类约束,给一段 canonical code / commit message / PR 描述,让模型按例模仿。
仅对**禁止性约束、不变量、阈值**写显式规则。

所有示例 MUST 自身不违反文档其他规则。

### 4.4 强制思维链与 Human-in-the-loop

- **强制思维链**:对复杂决策,要求模型在指定 tag(如 `<thinking>`)内列出评估维度后再下结论。引入的自定义 tag MUST 在文档内定义其语义与边界。
- **HITL 触发条件**(具体化,避免"重大决策"这种模糊词):
  - 写入生产数据库 / 外部 API 副作用
  - 任何删除操作
  - 修改超过 N 个文件(团队定 N)
  - 规则之间冲突且无法裁决
  - 用户意图存在 ≥2 种合理解读

## 5. 不写什么(噪音)

- **同义复述**:同一规则换词再说一遍。
- **蕴含信息**:已规定首选项就不必再列禁项。
- **过期 / 临时状态**(见 §2.3)。
- **装饰性内容**:导览语("下面我们将介绍…")、寒暄、装饰性 ASCII 美化。
  - 区分:**结构化** ASCII(目录树、状态机、拓扑图)信息密度高,保留;**装饰性** ASCII(花框、分隔线、ASCII art)删掉。
  - 判据:能用纯文本同样准确表达的 ASCII 是装饰;表达拓扑/状态/关系的 ASCII 是结构化。

> 模型生成文档时,会让当前会话中显著的概念在多个章节反复出现。写完通读一遍专门查这个。

---

## 6. 冲突与优先级

多条规则冲突时,文档 MUST 提供裁决路径。

默认优先级模板(可被项目覆盖):

**安全 / 合规 > 正确性 > 可维护性 > 性能 > 风格**

无法裁决时,触发 §4.4 HITL,而非自行选择。

---

## 7. 活文档:演化与维护

AGENTS.md 是活文档,不是宪法。

- **反模式条目 SHOULD 附溯源**:关联引发该规则的 commit / PR / issue,让后人能判断是否仍然相关。
- **定期 review**(建议每季度或每次大重构后):
  - 工具已能堵住的 → 删掉,迁移到 lint / CI。
  - 模型不再犯的 → 删掉。
  - 频率极低的 → 下沉到对应子目录的 AGENTS.md。
- **每条规则都在花 token 预算**:留下的必须配得上其位置。

## 自查清单

写完或修改后逐项核对:

1. 是否通过 §2 的三个排除测试?
2. 每条规则的 What 是否清晰?Why 在不显然时是否给出?How 是否可执行?
3. MUST / SHOULD / MAY 是否按 §4.1 的判据使用,有无滥用 MUST?
4. 关键词是否为行业惯用术语,有无生造词?中英文策略是否一致?
5. 是否存在冗余(同义复述、蕴含信息、装饰内容、过期状态)?删掉后含义无变化的 → 删。
6. 自定义 tag 是否在文档内定义语义?
7. 每个 ✅ / ❌ 示例是否准确演示规则,且**不违反文档其他规则**?
8. 规则冲突时是否有裁决路径,或回落到 HITL?
