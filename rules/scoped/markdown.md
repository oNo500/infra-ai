# Markdown

## 结构

- 标题树按 MECE 组织：兄弟节互斥、并集穷尽；「其他」类内容超过两成说明分类失败，重新划分
- 仅一个 H1（文件标题）；标题逐级递进，不跳级（H2 → H4 是错误）
- 重要信息前置，不写导览语（「下面我们将介绍…」）
- 密集信息用列表不用长段落；MUST NOT 使用表格，一律用列表替代
- 一篇一主题，命中两个主题就拆分

## 格式

- 代码块必须带语言标签；前后对比用 `diff` fence
- `**bold**` 不用 `__bold__`；`*italic*` 不用 `_italic_`；命令、路径、标识、值用行内 code
- `---` 只用于重大分节，不夹在标题之间当装饰
- GFM alerts 每篇 0–2 个，只给关键边界提醒；面向 AI 阅读的规则文档不用 alerts

  ```markdown
  > [!WARNING]
  > 该操作不可逆，执行前先备份。
  ```

- 会打断阅读流的长参考内容收进 `<details>`

  ```markdown
  <details>
  <summary>完整示例</summary>

  正文（summary 后必须空一行）。

  </details>
  ```

## 图示

- 表达架构、流程、关系优先 Mermaid

  ```mermaid
  graph TD
      A[Input] --> B{Decision}
      B -->|Yes| C[Action]
      B -->|No| D[Skip]
  ```

- 结构化 ASCII（目录树、状态机、拓扑）信息密度高，可用；MUST NOT 使用装饰性 ASCII（花框、分隔线、ASCII art）

## 面向检索与 AI 消费

- 章节自包含：不写悬空引用（「如上所述」「前文提到」），关键术语首次出现时简要定义——章节被单独检索出来也能独立理解
- 代码三件套：代码块前 1–2 句说明用途，代码完整可运行（伪代码明确标注），关键步骤带注释
- 关键信息不能只存在于图片里：图片必须配文字说明；内容形式优先级：列表 → Mermaid → 文字描述 → 截图
- 文档类型不混写（Diátaxis）：教程、操作指南、参考、解释同主题分篇，不在一篇里塞两种风格

## 术语

- 概念先行（ISO 704）：先明确概念，再找行业标准词直接采用；不确定时停下查证
- 一概一词：同一概念在一篇文档内不换词（不「聚合/聚合体/聚合对象」混用）
- 中英锚定只在首次出现：`aggregate（聚合）`，此后统一用一种形式
- 工程语境惯用的英文术语保留原词（`idempotent`、`debounce`），不强译、不造半英半中词
- 约束用「约定」，描述现状用「结构」——目录树写「目录约定」才构成约束
