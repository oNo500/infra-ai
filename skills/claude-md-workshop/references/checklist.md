# CLAUDE.md 审查清单（真实度 + 反模式）

> 审查模式的主引擎。来源：`ai-practice-CLAUDEmd元规范.md` 真实感自检 + `ai-reference-CLAUDEmd准则.md` §6 反模式。

## 怎么用

审查目标文件时，逐项过一遍。每项输出 `✓ 通过 / △ 警告 / ✗ 失败` + 一行说明。

## A. 真实度（元规范三问）

对每条规则都要问。任一条答「否」就标记问题。

- **A1**：我能说出这条规则对应的**一次具体犯错**吗？
- **A2**：这条能被 lint / typecheck / test / hook 验证吗？能验证的规则应该迁出去（验证 > 规则）。
- **A3**：这条是来自**真实失败沉淀**，还是**读了某篇博客**？博客抄的八成没必要。

## B. 长度与结构

- **B1 行数**：`wc -l` ≤ 200（硬建议）。> 300 必拆。50-100 = 健康区间。
- **B2 首尾效应**：前 5 行有绝对规则吗？末 3-5 行**重复**了吗？
- **B3 章节顺序**：绝对规则 → 摘要 → 命令 → 风格 → 工作流 → 保护区 → 重复。不符合则指出。
- **B4 Things That Will Bite You 段**：有没有？这段是整份 CLAUDE.md 里信号密度最高的。没有 → 建议补。

## C. 措辞

- **C1 NEVER 数量**：`grep -cE "^- NEVER" <file>` ≤ 5 条。超出稀释权重。
- **C2 模糊词**：`grep -iE "try|consider|maybe|might|could|建议|可考虑" <file>` 找到的都是问题。
- **C3 正向形式**：否定句数量是否过多？`grep -cE "Do NOT|Don't|Never use" <file>` > 5 → 可能有正向重写空间（NEVER 安全红线除外）。
- **C4 形容词污染**：`grep -iE "clean|readable|good|nice|appropriate" <file>` 命中 → 换数字或可验证标准。

## D. 指令完整性

每条规则都要有「行为 + 范围 + 失败模式」三要素。

- **D1 行为具体**：规则写了做什么或不做什么，且可验证？
- **D2 有 scope**：规则限定了文件路径 / 任务类型 / 条件？没有则容易跨上下文误用。
- **D3 有失败模式**：Claude 无法执行时知道停下而非瞎编？

## E. 反模式扫描（九类，对应 principles.md §6）

- **E1 冲突指令**：有没有同一行为被不同规则约束？`grep -rE "always|never" <file>` 找重复主题。
- **E2 过度冗长**：已在 B1。
- **E3 位置不当**：已在 B2。
- **E4 Vague Constraints**：已在 C4。
- **E5 No Failure Mode**：已在 D3。
- **E6 No Scope**：已在 D2。
- **E7 Kitchen Sink**：有没有项目介绍 / 完整依赖列表 / 部署文档 / 环境变量清单？这些不该在 CLAUDE.md。
- **E8 Fighting the Model**：有没有 `Never explain / Never ask / Never suggest alternatives`？这类压制默认行为的写法会浪费 token。
- **E9 Over-specified Code Style**：有没有 prettier 级细节（缩进 / 引号 / 尾逗号）？这些该交给 formatter。

## F. 载体归属

- **F1**：哪些规则应该迁 **hook**？（被忽视 ≥ 3 次、必须每次都发生的）
- **F2**：哪些规则应该迁 **lint/test**？（能被代码验证的）
- **F3**：哪些规则应该迁 **skill**？（多步流程、长参考材料）
- **F4**：哪些规则应该迁 **settings.json**？（env / permissions）

## G. 打分评级

综合 A-F 结果给总评：

- **骨架期**（< 50 行，前 3 段齐全）：尚未沉淀，无法全面审查。鼓励进入试用期。
- **试用期**（50-150 行，真实度 A 高）：健康成长中，列警告但不必大改。
- **稳定期**（50-150 行，A-F 全部 ≥ 70% 通过）：理想态。只需微调。
- **超重**（> 200 行 或 NEVER > 5 条）：需要裁剪 + 载体升级。
