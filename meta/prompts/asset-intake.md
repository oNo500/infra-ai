# 资产引入契约

用户给一个官方链接，要求把链接里的资产引入本仓。你在执行这个引入流程，
输出是账条目（`skills.json`/`meta/rules/<name>.md`）与提示文本，不是代码。
读完本文件再动笔。

## 触发

用户提供一个官方链接（参考来源）并要求引入其中的资产——例如「把这个
页面里的东西加进来」「参考这个文档建个 skill」。链接即 `refUrl` 的取值来源。

## 抓取纪律

- MUST 用真实工具调用（WebFetch 等）抓取该链接，读到实际返回内容再往下走
- MUST NOT 凭记忆或训练数据描述页面内容——即使你「认识」这个页面，
  内容可能已变，训练数据不是当前状态的证据
- 抓取失败（网络错误、404、需要登录墙等）即停：报告失败原因，等用户决定
  下一步，不要猜测页面内容后继续判类

## 提取与判类

抓到页面后，列出页面提供或推荐的全部资产，逐个判类：

- **操作性知识 → skill**：页面教的是「怎么做一件事」（工作流、工具用法、
  可执行的操作序列）。按 `SKILLS.md` 现有三分法判 `official`/`mirror`/`custom`，
  在 `skills.json` 建对应条目
- **姿态化约束素材 → rule**：页面表达的是「应该/不应该怎样」的立场或
  约束，不是操作步骤。在 `meta/rules/<name>.md` 建或更新元指令，
  `status: stub` 起步（除非素材已完整到可以直接写 `ready`）
- **mcp / agent → 不建账**：本仓当前不追踪这两类资产（mcp 只有
  `docs/mcp/*.md` 说明，agent 连 kind 都不存在）。发现即输出提示，
  不创建任何文件：

  ```
  发现 <名称>，类别 <mcp|agent>，账未立项，refUrl=<链接>
  ```

  由用户决定是否为这两类立项，MUST NOT 自行建账或抢先设计 schema

## 溯源落账

两层溯源缺一不可，字段定义见 `SKILLS.md`（skill）与 `meta/README.md`
元指令格式（rule）：

- `refUrl`——参考来源，回答「官方怎么说」。取值是用户给的链接，或者
  该链接页面内指向的更精确的权威子页（例如落地页链到具体的
  getting-started 页，后者更精确就记后者）
- 实际来源——回答「东西从哪来」，按情况二选一：
  - `repo`/`path`/`commit`（mirror 类：从某个 repo 的某个目录拉取）
  - `install`（自由文本：实际来源是命令或目录，如工具自带的
    skill 目录）；MUST 仅记录该命令/路径，MUST NOT 执行它
- rule 类另在元指令正文首段写人读的收编来源叙述（既有纪律不变，
  `refUrl` 是给链接健康检查用的机器可读补充，不能取代人读叙述）

示例（真实案例）：ai-sdk 的 coding-agents 页面
`https://ai-sdk.dev/docs/getting-started/coding-agents` 是 refUrl；
页面实际推荐的资产在 `vercel/ai` 这个 repo 里，落 `repo: vercel/ai`
（配 `path`/`commit`），两个字段并存，缺一不成立。

## 人审

全部产出——新建或修改的账条目、元指令文件——MUST 提交用户确认后才落地。
引入 `official` skill 时，安装方式按 `SKILLS.md` 现行方式（`pnpx skills add`），
不新造安装路径。

## 来源更新（独立步骤）

链接健康检查报告某资产的 `refUrl` broken 或 moved 后，更新 `refUrl`
是与检查解耦的独立步骤，两种走法：

- 人工直接改账（改 `skills.json` 条目或 rule frontmatter）
- 指派 AI 重新检索定位新的权威页：按本契约的抓取纪律（真实抓取、
  禁凭记忆）验证新地址确实有效后，再更新 `refUrl`

moved 且检查结果附了新地址（重定向目标）时，优先验证这个新地址，
不要另起炉灶搜索。

## 非目标

- mcp/agent 不建账，只提示——立项是另一件事，不在本契约范围内
- 不自动改写/自动修复 `refUrl`——健康检查只报告，更新永远是本节
  描述的独立步骤
- 不执行 `install` 字段里记录的命令——它是文档，不是脚本
